Конвейер из Агентов:
* PM - дает задание другому агенту. Например: BA проанализируй вот такую идею и выдай Requirements Increment. Анализирует инкремент и принимает финальное решение об изменениях. Другой пример: ARCHITECT, сделай такое-то изменение в архитектуре и дай задачу DEVELOPER'у на необходимые изменения в коде.
* BA - анализирует бизнес-потребности и переводит их в требования.
* ARCHITECT - создает архитектуру приложения и другие правила, которым должен следовать код
* DEV-TASKS - тех. аналитик, отлавливает изменения в требованиях и создает для них задачи для DEVELOPER
* DEVELOPER - реалирует код по файлу DEV-Tasks, создает unit tests, etc. Комитит работающий код
* TESTER

# Целевой конвейер
1. Intake / триаж (что за задача, в какой конвейер идет)
2. Анализ требований (создание тех. задач в файл)
3. Импакт-анализ и дизайн (архитектор, программист)
4. Имплементация
5. Стек верификации (бирюзовая) — типы, тесты, SAST, регрессия
6. Ревью и compliance (жёлтая) — человек, PII/PCI, аудит-след
7. Merge → CI/CD

Guardrails:

## Фаза 1. Сначала верификация, потом агентская разработка
Концепт, как отказаться/уменьшить ручное тестирование:
* Все, что связано с тест-кейсами - повторяемое+детерминированное тестирование - все в unit/e2e тесты! Все текущее ручное детерменированное тестирование - в полную автоматизацию, в правильную пирамиду!
* На переходном периоде остается полное ручное тестирование! Оно будет дублировать автоматическое и цель ручного тестирования - ничего не найти!
* Правильно для сверх-важных областей (платежи, compliance, calculations) - ручное детерменированное тестирование остается навсегда!
* а вот exploratory testing, UX, UAT - это остается человеку и в Фазу 1 не входит - **эти виды верификации остаются в конвейере на человеке**.
* code review - человеком, поэтому каждый инкремент должен быть не огромным, чтобы человек легко мог проверить

? где место test-cases? Ответ-дискуссионный: детерменированные - сразу в коде, остальные можно хранить в confluence. И чтобы человек мог в любой момент прочитать все тесты из кода:
* нужен генерируемый отчет, который по коду генерит словесные тест-кейсы
* сами тесты пиши на Gherkin

**Выход фазы**: на пилотных задачах любое изменение проходит через детерминированный гейт (тесты, compliance, регрессия, ответственное code review)
