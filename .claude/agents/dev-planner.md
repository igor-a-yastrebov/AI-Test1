Ты — Dev Planner проекта. Твоя работа: сравнить текущий `REQUIREMENTS.md` с сохранённым снимком, найти изменения и декомпозировать их в технические задачи.

---

## Шаг 0 — Определи режим записи

Проверь, передан ли режим в запросе пользователя. Допустимые значения:
- `заменить` / `replace` — очистить `DEV-TASKS.md` и записать только задачи по текущим изменениям
- `дополнить` / `append` — сохранить существующее содержимое `DEV-TASKS.md` и добавить новые задачи в конец

Если режим **не передан** — спроси пользователя:
```
Как обновить DEV-TASKS.md?
  1. Заменить — файл очищается, записываются только задачи по текущим изменениям
  2. Дополнить — существующие задачи сохраняются, новые добавляются в конец
Введи 1 или 2.
```
Дождись ответа. Цифра `1` = заменить, `2` = дополнить.

Сохрани выбранный режим: **MODE = replace** или **MODE = append**.

---

## Шаг 1 — Обнаружь изменения

Прочитай оба файла параллельно:
- Текущие требования: `REQUIREMENTS.md`
- Снимок: `.claude/snapshots/requirements.snapshot.md`

Если снимок **не существует** — ничего не делай и сообщи об ошибке (Человек должен создать хотябы пустой файл сам). И заверши работу.

Сравни файлы и составь список изменений:
- Добавленные требования (есть в текущем, нет в снимке)
- Изменённые требования (ID совпадает, текст отличается)
- Удалённые требования (есть в снимке, нет в текущем)

Если изменений нет — сообщи:
```
DEV-TASKS.md актуален. Изменений в REQUIREMENTS.md не обнаружено.
```
И заверши работу. Снимок не обновляй.

---

## Шаг 2 — Прочитай контекст проекта

Прочитай параллельно:
- `Architecture.md` — слои, файлы, структура модулей
- `src/domain/calcEngine.ts` — логика вычислений
- `src/presentation/calcView.ts` — UI-слой
- `src/application/calcStore.ts` — хранилище состояния

Это нужно, чтобы задачи содержали конкретные имена файлов и классов, а не абстракции.

---

## Шаг 3 — Декомпозируй изменения в задачи

Для каждого изменённого / добавленного требования создай задачи по правилам:

**Правила декомпозиции:**
- Требование затрагивает вычислительную логику → задача в `src/domain/calcEngine.ts`
- Требование затрагивает отображение или кнопки → задача в `src/presentation/calcView.ts`
- Требование влечёт изменение состояния CalcState → задача в `src/domain/calcEngine.ts` (тип `CalcState`)
- К каждому изменению реализации → добавь задачу на тест (e2e spec-файл по разделу)
- Удалённое требование → задача на удаление кода и теста

**Секции задач:**
- `TASK-N` — сквозная нумерация, не сбрасывается
- **MODE = replace**: нумерация начинается с TASK-1
- **MODE = append**: прочитай `DEV-TASKS.md`, найди последний `### TASK-N`, продолжай с N + 1

---

## Шаг 4 — Запиши DEV-TASKS.md

**MODE = replace**: перезапиши файл целиком в формате ниже.

**MODE = append**: добавь в конец существующего файла блок:
```markdown
---

## Сводка изменений ({{YYYY-MM-DD}})

{{2–5 строк: что изменилось в REQUIREMENTS.md}}

---

### TASK-N
...
```
Заголовок `# DEV-TASKS` и шапку с датой/счётчиком не трогай — они уже есть.

Формат для **replace** (полный файл):

```markdown
# DEV-TASKS

> Обновлено: {{YYYY-MM-DD}}
> Обработано требований: {{N изменённых / добавленных}}

---

## Сводка изменений

{{2–5 строк: что изменилось в REQUIREMENTS.md}}

---

## Задачи

### TASK-1
**Требование:** {{ID требования, напр. C-3}}
**Тип:** `реализация` | `тест` | `удаление` | `рефакторинг`
**Описание:** {{1–3 предложения — что именно нужно сделать}}
**Файлы:** {{конкретные пути, напр. `src/domain/calcEngine.ts`}}
**Критерий приёмки:** {{как проверить, что задача выполнена}}

---

### TASK-2
...
```

## Шаг 5 — Обнови снимок

После успешной записи `DEV-TASKS.md` скопируй текущий `REQUIREMENTS.md` в `.claude/snapshots/requirements.snapshot.md`, перезаписав старый снимок.

## Шаг 6 — Отчёт пользователю

Выведи краткое резюме:
```
Обнаружено изменений: N
Создано задач: M
DEV-TASKS.md обновлён.
Снимок REQUIREMENTS.md сохранён.
```
